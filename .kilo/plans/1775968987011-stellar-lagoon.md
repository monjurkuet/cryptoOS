## Goal

Reduce unnecessary growth of the MongoDB `events` collection while preserving the data needed for APIs, signal generation, and debugging.

## What I checked

- `events` is a catch-all raw event log written by the lifecycle storage handler for every event except `leaderboard`.
- This happens in `market-scraper/src/market_scraper/orchestration/lifecycle.py` via `_subscribe_storage_handler()`.
- The same handler also materializes data into dedicated collections:
  - `signals`
  - `trader_positions`
  - `trader_current_state`
- The repository also supports querying `events` directly via:
  - `store()` / `store_bulk()`
  - `query()`
  - `get_latest()`
  - `aggregate_ohlcv()`
- Current retention in `market-scraper/config/market_config.yaml` keeps `events` for 7 days.
- The codebase describes `events` as a raw audit log, but much of its content is duplicated into more specific collections.

## Findings

### Why `events` becomes huge

1. Every emitted event is persisted to `events`, including high-frequency market/trader events.
2. Some event types are also persisted again in specialized collections, creating intentional duplication.
3. `aggregate_ohlcv()` currently derives candles from raw `events` trade rows instead of relying only on candle collections.
4. `query()` and `get_latest()` still make `events` the generic source of truth for raw event inspection.
5. Buffer settings in YAML currently show `flush_interval: 50.0`, which reduces write frequency but does not reduce the number of raw events stored.

### Is it inefficient / bad code?

Partly yes.

- Keeping a short-lived raw event log is standard for observability and replay.
- Persisting all event types without a storage policy is inefficient once dedicated collections already exist.
- The current setup mixes two goals:
  - audit/debug log
  - application data store
- That makes `events` too large for its real value.

### Safe conclusion

`events` should not remain the default persistence target for every event type. It should become a controlled audit stream, not the primary storage path for data already materialized elsewhere.

## Recommended target design

### Principle

Store each datum once in its canonical collection. Keep raw events only where they provide unique debugging or replay value.

### Canonical collections

- Candles -> symbol candle collections
- Trader position history -> `trader_positions`
- Latest trader state -> `trader_current_state`
- Signals -> `signals`
- Trader score history -> `trader_scores` when enabled
- Leaderboard snapshots -> `leaderboard_history`

### Raw `events` collection should keep only

Recommended retained raw event types:

- `trading_signal` for audit/debug
- selected on-chain / external connector events if they are not otherwise materialized
- optionally failures / dead-letter references only

Recommended excluded raw event types:

- `trader_positions` because already stored in `trader_positions` and `trader_current_state`
- candle / market stream events that already land in dedicated candle collections
- any other event type already persisted in a canonical collection and not needed for replay

## Implementation plan

### 1. Introduce an explicit raw-event persistence policy

Update the lifecycle storage handler so it does not blindly persist every event to `events`.

Implementation direction:

- Add a helper in `LifecycleManager` such as `_should_store_raw_event(event)`
- Use an allowlist instead of a skiplist
- Default to storing only low-volume, audit-worthy event types

Initial recommended allowlist:

- `trading_signal`
- any connector events not otherwise materialized

Initial recommended denylist:

- `trader_positions`
- candle/market stream events already materialized elsewhere
- `leaderboard` remains excluded

### 2. Stop using raw `events` as a dependency for core product data

Review repository/API paths that rely on `events` and either:

- keep them only for debug/admin use, or
- move them to canonical collections

Most important item:

- `aggregate_ohlcv()` currently aggregates from `events` trade rows. If the system already stores dedicated candles and primarily serves candle history from candle collections, this method should either be deprecated or clearly marked debug-only.

### 3. Tighten retention for `events`

After the raw-event policy change:

- reduce `storage.retention.events` from `7` to `1` or `2` days
- keep dead letters separately for failure analysis

This is a low-risk storage win once raw persistence is narrowed.

### 4. Add observability around raw event volume

Add metrics/logging for:

- number of raw events persisted by type
- number of raw events skipped by policy
- collection growth stats if available in health/status

This prevents the collection from silently ballooning again.

### 5. Add tests for the policy

Test cases should cover:

- `trader_positions` does not write to raw `events` but still materializes current state/history
- `trading_signal` still persists correctly
- denied event types are skipped from raw storage
- allowlisted event types still land in `events`

## Proposed execution order

1. Inspect exact event types emitted by collectors/processors.
2. Classify each event type as one of:
   - canonical-only
   - raw-audit-only
   - both
3. Implement `_should_store_raw_event(event)` in lifecycle storage handling.
4. Keep materialization of specialized collections unchanged.
5. Reduce `events` retention after policy is in place.
6. Add/update tests for lifecycle storage behavior.

## Recommended default policy

For production efficiency, the best default is:

- do **not** store high-volume market/trader state events in `events`
- keep `events` as a small audit/debug stream only
- rely on canonical collections for all product/API reads
- reduce raw event TTL to 1-2 days

## Risks and tradeoffs

### Pros

- much smaller `events` collection
- lower write amplification
- clearer data ownership per collection
- less duplication and lower storage cost

### Cons

- less raw replay capability for excluded event types
- any admin/debug functionality depending on `events` may need adjustment
- if some processor silently depends on raw historical `events`, that dependency must be identified first

## Implementation note

This should be done as a clean-slate production change, not a compatibility-preserving migration. That matches the repo instructions and is the right approach here.
